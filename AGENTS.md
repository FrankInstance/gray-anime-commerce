# Gray Project Agent Notes

## Product Voice

- 面向用户的页面文案要直接、克制、服务当前操作，避免运营增长口吻。
- 不要在用户界面展示“增加阅读量”“拉新”“转化”“增长”等运营目标或解释性文案。
- 不要把内部技术动作暴露给用户，例如“生成支付订单”“创建支付单”“生成订单中”。
- 不要用无意义的英文眉标装饰充值、会员、积分等页面，例如 `POINTS · VIP`。
- 充值、支付、会员、积分相关入口优先使用用户能理解的短文案，例如“充值”“选择”“支付”“确认”“详情”“待支付”“处理中”。

## Encoding Safety

- 处理中文文本、SQL 种子数据、章节正文、页面文案时，必须保持 UTF-8 编码，提交前要抽样检查实际文件和运行中数据是否为正常中文。
- 不要通过可能改变编码的 PowerShell 命令字符串、管道或 shell 拼接方式直接把中文写入数据库；需要写入 MySQL 等服务时，优先使用 UTF-8 文件导入、参数化脚本，或 UTF-8 十六进制字面量等不会产生乱码的方式。
- 如果终端输出出现 `å¤œ...`、`闈㈠...` 等疑似 mojibake，不能把它当作正常结果继续推进，必须先确认文件编码、接口响应和数据库实际内容。
- 修改包含中文的种子数据后，至少用浏览器同等的 UTF-8 解析方式验证接口返回预览，确认页面不会展示乱码。

## Production Readiness

- 项目已进入收尾阶段。后续实现默认按可上线的正式项目标准设计，不用仅能演示的临时方案替代完整的安全性、数据一致性、错误处理和测试。
- 本地模拟能力必须与正式接口边界分离并明确标注；不能让模拟支付、开发令牌、默认账号或调试入口在生产配置中自动启用。
- 密码、私钥、访问令牌和第三方密钥不得提交到 Git。开发密钥只能由脚本在本地生成，生产密钥必须从部署平台 Secret 或密钥管理服务注入。
- 身份与权限必须在服务端验证，不能信任前端状态、客户端身份头或仅由 UI 隐藏按钮形成的权限限制。
- 涉及认证、订单、支付、库存和用户资产的修改，必须补充失败路径与越权路径测试，并运行核心回归后再提交。

## Maven Repository

- 在本项目运行 Maven 时，统一显式使用 `%USERPROFILE%\.m2\repository` 作为本地仓库，例如 `mvn -Dmaven.repo.local="$env:USERPROFILE\.m2\repository" test`；不要使用 Maven 安装目录下的 `D:\apache-maven-3.9.11\maven_repo`，避免权限错误和缓存位置不一致。
- Docker 镜像通过 `backend/*/target/*.jar` 复制服务产物。修改 Java 源码后，在执行 `docker compose up --build` 前必须先运行 Maven `package`，并确认 JAR 的更新时间不早于源码。
